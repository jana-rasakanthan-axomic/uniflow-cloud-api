# EP01-T12: STS Credential Issuance & Direct Upload

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** POST /edge/upload/sts
**Depends On:** EP01-T01, EP01-T02, EP01-T09, EP01-T10

## Summary

Implement scoped STS credential issuance for edge agents to upload files directly to S3 without transiting through the cloud broker. Also implement the upload progress reporting endpoint where agents report chunk-level progress for real-time dashboard updates. Files never pass through the UniFlow cloud infrastructure, satisfying the core data privacy requirement.

## Context

BR-01 forbids storing raw files in UniFlow infrastructure. The solution is scoped STS credentials that grant the edge agent temporary, file-specific write permission to a designated S3 bucket. The credentials expire after 12 hours and are scoped to specific object keys (file paths). The edge agent uploads directly to S3, and only progress metadata is sent to the cloud broker. This architecture ensures that even if the cloud broker is compromised, it cannot access file contents.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /edge/upload/sts accepts JSON payload with job_id and file_ids array
- [ ] Endpoint validates that the requesting agent is authorized to upload these files (agent owns the device that scanned them)
- [ ] STS credentials issued with 12-hour expiry and scoped to specific S3 object keys
- [ ] Response includes access_key, secret_key, session_token, expiry timestamp, and S3 bucket/key pairs
- [ ] Edge agent uploads files directly to S3 using multipart upload for files >10MB
- [ ] POST /edge/upload/progress accepts JSON payload with job_id, file_id, chunks_completed, total_chunks, status
- [ ] Progress updates stored in job_files table for Curator dashboard visibility
- [ ] File status transitions from PRE_REGISTERED → UPLOADING → SYNCED on completion
- [ ] Hash verification performed by S3 (ETag check) — no cloud broker involvement
- [ ] Failed uploads automatically trigger ghost record cleanup (oa_asset_id deleted from OA)
- [ ] Files never transit through the cloud broker (no file bytes in HTTP requests/responses)

## Business Rules

- BR-01: No Cloud File Storage

## BDD Scenarios

- Raw files never transit through UniFlow cloud during upload
- Expired STS token refreshed silently during resume

## Technical Notes

Use AWS STS AssumeRole with a policy that restricts PutObject to specific object keys (e.g., `uploads/{org_id}/{job_id}/{file_id}/*`). The 12-hour expiry is long enough to cover overnight uploads but short enough to limit exposure if credentials are stolen. The edge agent must handle expired credentials gracefully by requesting fresh STS credentials via this endpoint. S3 multipart upload provides automatic resume capability — the edge stores the upload_id in SQLite and can continue appending chunks after interruption.
