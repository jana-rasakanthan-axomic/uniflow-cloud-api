# EP01-T10: Job State Machine & Transitions

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** PATCH /jobs/{id}/state
**Depends On:** EP01-T01, EP01-T02, EP01-T09

## Summary

Implement the job state machine with strict transition rules enforced at both the application layer and database layer. Support user-initiated actions (pause, resume, cancel) and automatic transitions (consent, completion, timeout). Track file-level states independently. Implement a background task to timeout jobs in WAITING_FOR_AGENT state after a configurable period (default 7 days).

## Context

The job state machine is the contract between Curator and Contributor. It must prevent illegal transitions (e.g., resuming a DENIED job) and provide deterministic behavior (BR-14b). File-level states are independent because a single job may span multiple devices with different availability (BR-15a). The 7-day timeout ensures that stale requests do not linger indefinitely, while remaining long enough to cover a full work week plus weekend (BR-15b).

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] PATCH /jobs/{id}/state accepts JSON payload with action (cancel, pause, resume)
- [ ] Application validates transitions against state machine rules (PRE_REGISTERING → WAITING_FOR_AGENT or FAILED, etc.)
- [ ] Database CHECK constraint enforces valid job states
- [ ] Invalid transitions return 409 Conflict with descriptive error message
- [ ] Pause action transitions IN_PROGRESS → PAUSED_USER
- [ ] Resume action transitions PAUSED_USER → IN_PROGRESS
- [ ] Cancel action transitions WAITING_FOR_AGENT, IN_PROGRESS, or PAUSED_USER → CANCELLED
- [ ] Agent consent transitions WAITING_FOR_AGENT → IN_PROGRESS
- [ ] Agent denial transitions WAITING_FOR_AGENT → DENIED
- [ ] Terminal states (DENIED, TIMEOUT, COMPLETED, PARTIALLY_FAILED, FAILED, CANCELLED) reject all outbound transitions
- [ ] File-level states tracked independently in job_files table
- [ ] Background task runs every hour to find jobs in WAITING_FOR_AGENT state older than timeout period (default 7 days, org-configurable)
- [ ] Expired jobs transition to TIMEOUT (not FAILED)
- [ ] Curator sees "Expires in X days" on queued jobs
- [ ] Curator can trigger "Re-send request" on TIMEOUT jobs to create a new job

## BDD Scenarios

- Job transitions through happy path to completion
- Contributor denies upload request (transitions to DENIED)
- Contributor pauses and resumes an active job
- Contributor cancels a job during upload
- Some files fail during upload (transitions to PARTIALLY_FAILED)
- System rejects illegal state transition
- System rejects transition from terminal state
- Files within a job track independent states
- Job times out after default period
- Curator sees expiry countdown on queued jobs
- Organization configures custom timeout period

## Technical Notes

Use an in-code state machine class with explicit transition methods and a validation layer. The database CHECK constraint is defense-in-depth (ADR-0002) — the application should never send an invalid state to the database. The timeout background task must be idempotent (safe to run multiple times on the same job). File states must support independent transitions (one file UPLOADING while another PAUSED). Ghost cleanup on timeout must delete OA placeholder records for files that were pre-registered but never uploaded.
