# EP01-T02: Three-Router Architecture & Rate Limiting

**Epic:** EP-01 Platform Foundation
**Points:** 2
**Primary Endpoint:** /api/v1/* (router groups)
**Depends On:** EP01-T01

## Summary

Organize the FastAPI application into three logically isolated router groups with independent rate limiting counters. Auth router handles authentication, web router serves the Curator dashboard, and edge router serves machine-to-machine signaling. This isolation prevents cross-domain attacks and ensures that an edge flood does not degrade web dashboard performance.

## Context

Enterprise AEC environments experience network events that cause thousands of agents to reconnect simultaneously. Without router isolation, aggressive rate limiting on edge endpoints would block Curator dashboard access (BR-26). Independent rate limit counters ensure that a 5,000-agent reconnection flood does not prevent Curators from viewing job status.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 2.2

## Acceptance Criteria

- [ ] Auth router mounted at `/api/v1/auth/` with endpoints for device linking and token refresh
- [ ] Web router mounted at `/api/v1/` with endpoints for jobs, devices, collections, and audit logs
- [ ] Edge router mounted at `/api/v1/edge/` with endpoints for state reporting, polling, STS credentials, and upload progress
- [ ] Independent rate limiting middleware for each router group (auth: 20 req/min/IP, web: 120 req/min/user, edge: 60 req/min/agent)
- [ ] Rate limit counters are isolated per router group (edge flood does not affect web or auth counters)
- [ ] 429 responses include Retry-After header with recommended backoff duration
- [ ] Auth router validates JWT bearer tokens for protected endpoints
- [ ] Web router requires user-scoped JWT tokens (not device tokens)
- [ ] Edge router requires device-scoped JWT tokens (not user tokens)
- [ ] Security log entry created when web session attempts to call edge endpoint or vice versa

## Business Rules

- BR-26: Three-Router Architecture

## BDD Scenarios

- Auth endpoints are isolated from web and edge routers
- Edge flood does not affect web dashboard
- Web endpoint cannot spoof edge heartbeat

## Technical Notes

Use FastAPI's APIRouter for logical separation. Rate limiting can be implemented via middleware that checks the router group prefix. Store rate limit counters in-memory using a sliding window algorithm (no Redis per BR-07). JWT token validation should check the token's intended audience claim to prevent cross-router token reuse. The security isolation is critical for preventing web UI compromise from being leveraged to send spoofed edge heartbeats.
