# EP01-T09: Job Creation & OA Pre-Registration

**Epic:** EP-01 Platform Foundation
**Points:** 5
**Primary Endpoint:** POST /jobs
**Depends On:** EP01-T01, EP01-T02, EP01-T03, EP01-T04

## Summary

Implement upload job creation with OpenAsset pre-registration. When a Curator creates a job, the cloud broker creates a job record in PRE_REGISTERING state, then batches files into groups of 25 and pre-registers them with the OpenAsset API using a semaphore-controlled concurrency limit of 10. On success, the job transitions to WAITING_FOR_AGENT. On failure, ghost records are cleaned up and the job transitions to FAILED.

## Context

Pre-registration creates placeholder asset records in OpenAsset before the actual file upload begins. This reserves the oa_asset_id that will be referenced during the upload. The batching (25 files per request) balances HTTP overhead against failure blast radius (BR-18). The semaphore (max 10 concurrent requests) prevents UniFlow from overwhelming the customer's OpenAsset instance (BR-16b). Exponential backoff on 429 responses ensures well-behaved API client behavior (BR-16a).

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /jobs accepts JSON payload with collection_id and file_ids array
- [ ] Job record created in PRE_REGISTERING state
- [ ] Job_files records created for each file in DISCOVERED state
- [ ] Files are batched into groups of 25 for pre-registration API calls
- [ ] OA API calls routed through asyncio.Semaphore(10) to limit concurrency
- [ ] On 429 response, exponential backoff: 2s, 5s, 15s, 60s
- [ ] If Retry-After header present, use that value instead of default backoff
- [ ] After 4 retries, mark operation as RATE_LIMITED and alert Curator
- [ ] On successful pre-registration, oa_asset_id stored in job_files record and status updated to PRE_REGISTERED
- [ ] After all batches complete, job transitions to WAITING_FOR_AGENT
- [ ] On pre-registration failure, ghost records deleted via OA API (see BR-17)
- [ ] If ghost cleanup fails, failure logged for manual remediation
- [ ] On unrecoverable failure, job transitions to FAILED
- [ ] Response returns job_id and status on success (201)

## BDD Scenarios

- New job starts in PRE_REGISTERING state
- OpenAsset API fails during pre-registration (transitions to FAILED)
- Rate limit triggers exponential backoff
- Retry-After header overrides default backoff
- Concurrent OpenAsset API calls are limited
- Failed upload triggers ghost record cleanup
- Ghost cleanup failure is logged for remediation
- Pre-registration respects batch size limit
- Partial batch failure does not affect other batches

## Technical Notes

The OAClientService must be instantiated as a singleton with a class-level asyncio.Semaphore to ensure the concurrency limit applies across all simultaneous job creations. The semaphore is per-instance (per Fargate task) as documented in ADR-0003 — document scaling implications for ops. Ghost cleanup must be attempted even on partial failures (e.g., batch 2 of 4 fails). Use a retry counter in the job_files record to prevent infinite retry loops on persistent OA API errors.
