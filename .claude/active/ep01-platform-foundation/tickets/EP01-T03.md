# EP01-T03: Device Linking & Setup Codes

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** POST /auth/device/link
**Depends On:** EP01-T01, EP01-T02

## Summary

Implement the device linking flow where a Contributor enters an 8-character setup code to link their edge agent to a UniFlow organization. The endpoint validates the code, creates a device record, issues a JWT token pair (access + refresh), and logs the linking event to the immutable audit trail.

## Context

Device linking is the initial trust establishment between an edge agent and the cloud broker. The setup code is a time-limited single-use code generated by a Curator. Once linked, the device receives a refresh token that can be rotated indefinitely, eliminating the need for password-based authentication (BR-25). All linking events must be audited with source IP and machine metadata for enterprise security compliance.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /auth/device/link accepts JSON payload with setup_code, machine_name, and os fields
- [ ] Endpoint validates that the setup code exists, is not expired, and has not been used
- [ ] Setup code is marked as used (used_at timestamp updated) after successful validation
- [ ] New device record created with agent_id (UUID), machine_name, os, and initial status ONLINE
- [ ] JWT access token issued with 1-hour expiry and device-scoped claims (agent_id, org_id)
- [ ] JWT refresh token issued with 90-day expiry and stored in refresh_tokens table with chain_id and sequence_num 1
- [ ] Successful link event logged to audit_log with event_type DEVICE_LINKED, including source_ip, machine_name, and agent_id
- [ ] Failed link attempts logged to audit_log with event_type LINK_CODE_FAILED, including source_ip and attempt count
- [ ] Response returns agent_id, access_token, and refresh_token on success (200)
- [ ] Response returns 400 if setup_code is malformed
- [ ] Response returns 401 if setup_code is invalid or expired
- [ ] Response returns 410 if setup_code has already been used

## Business Rules

- BR-25: Immutable Audit Trail

## BDD Scenarios

- Device linking is logged in audit trail
- Failed link attempts are logged

## Technical Notes

The setup code lookup must be atomic (SELECT FOR UPDATE) to prevent race conditions if multiple agents try to use the same code simultaneously. The agent_id UUID must be cryptographically random (not sequential) to prevent enumeration attacks. The refresh token's chain_id is a UUID that identifies the token family — all tokens rotated from the original share the same chain_id. This enables chain revocation on reuse detection (see EP01-T04).
