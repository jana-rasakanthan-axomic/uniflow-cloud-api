# EP01-T07: Command Queuing & Delivery

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** POST /commands
**Depends On:** EP01-T01, EP01-T02, EP01-T06

## Summary

Implement command queuing for edge agents. When a Curator triggers a sync request or other command, the cloud broker queues it in the database and integrates with the SignalingService to wake any held long-poll connection for immediate delivery. If the agent is offline, the command remains queued and is delivered automatically on reconnect.

## Context

Commands are how Curators communicate with edge agents (upload requests, pause/resume, refresh scans). The command must be delivered near-instantly if the agent is actively polling (~50ms via BR-10), or queued server-side if the agent is offline (BR-12). The queuing mechanism must survive cloud broker restarts — commands are persisted in PostgreSQL, not held in memory.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /commands accepts JSON payload with agent_id, type (e.g., "upload_request", "pause", "resume"), and payload object
- [ ] Command record created in commands table with status PENDING
- [ ] Integration with SignalingService to set the asyncio.Event for the target agent_id (wakes held poll)
- [ ] If agent is actively polling, command is delivered within ~50ms
- [ ] If agent is offline or between poll cycles, command remains PENDING until next poll
- [ ] GET /edge/poll checks for pending commands and returns the oldest PENDING command first (FIFO)
- [ ] Delivered command is marked DELIVERED (status updated, delivered_at timestamp set)
- [ ] Commands older than 24 hours are marked EXPIRED and not delivered (background cleanup task)
- [ ] Response returns command_id and queued: true on success (201)
- [ ] Multiple commands for the same agent are queued and delivered in chronological order

## Business Rules

- BR-12: Offline Command Queuing

## BDD Scenarios

- Command delivered to actively polling agent
- Command delivered between poll cycles
- Command queued for offline agent and delivered on reconnect
- Multiple commands queued while agent is offline

## Technical Notes

The SignalingService must expose a dispatch_command method that looks up the agent's asyncio.Event (if any) and sets it. If no active poll connection exists, the command is simply queued in the database and will be picked up on the next poll. The FIFO ordering is critical to prevent newer commands from overtaking older ones. Use SELECT FOR UPDATE when popping pending commands to prevent race conditions. The 24-hour expiry prevents the queue from growing unbounded if an agent is permanently offline.
