# EP01-T06: Long-Poll Signaling Channel

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** GET /edge/poll
**Depends On:** EP01-T01, EP01-T02, EP01-T03, EP01-T04, EP01-T05

## Summary

Implement the long-poll signaling channel where edge agents hold an outbound HTTPS connection for up to 55 seconds (not 60 seconds — avoids ALB idle timeout race condition). If a command is queued during the hold, return it immediately (~50ms delivery). If no command arrives, return a no-action response at timeout. This provides near-instant command delivery when agents are actively polling, with zero database writes from idle connections.

## Context

The long-poll channel is the core of UniFlow's signaling architecture. It must support thousands of idle agents without proportional database load (BR-05). Traditional short-polling would create 600,000 database writes per hour from 10,000 agents polling every 60 seconds. Long-poll with asyncio.Event reduces this to ~3 writes per agent per day (ONLINE, state changes, OFFLINE). The 55-second hold avoids the ALB 60-second idle timeout race condition identified in Gemini review.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 2.3

## Acceptance Criteria

- [ ] GET /edge/poll accepts agent_id query parameter
- [ ] Endpoint validates JWT token and ensures agent_id matches device record
- [ ] Connection is held for up to 55 seconds using asyncio.Event.wait(timeout=55)
- [ ] If a command is queued during the hold, the event is set and the endpoint returns immediately
- [ ] Response includes command action and payload if available
- [ ] Response returns {action: "none"} if timeout occurs with no command
- [ ] Idle connections produce zero database writes (no heartbeat updates)
- [ ] In-memory tracking of active poll connections to exclude from stale detection
- [ ] Support for thousands of concurrent held connections without proportional memory/CPU growth
- [ ] Connection termination (client disconnect) cleans up in-memory state
- [ ] Response time for queued command is <100ms (typically ~50ms)

## Business Rules

- BR-05: Zero Database Load from Idle Connections
- BR-08: Firewall-Friendly Communication (outbound HTTPS only)
- BR-09: Long-Poll Hold Duration
- BR-10: Near-Instant Command Delivery

## BDD Scenarios

- Idle long-poll connection produces no database writes
- Long-poll connection held for 60 seconds when idle (updated to 55s)
- Long-poll returns immediately when command is queued
- Command delivered to actively polling agent
- Agent communicates using outbound HTTPS only

## Technical Notes

Use asyncio.Event per agent (stored in a dictionary keyed by agent_id). When a command is queued (see EP01-T07), set the event to wake the held connection. The 55-second timeout must be enforced at the application layer, not relying on ALB behavior. Memory usage for 10,000 agents with held connections should be <100MB (Event objects are lightweight). The endpoint must handle graceful shutdown by closing all held connections before the container terminates.
