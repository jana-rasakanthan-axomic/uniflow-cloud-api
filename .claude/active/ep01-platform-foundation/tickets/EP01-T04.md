# EP01-T04: Token Refresh & Reuse Detection

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** POST /auth/tokens/refresh
**Depends On:** EP01-T01, EP01-T02, EP01-T03

## Summary

Implement token refresh with automatic rotation and refresh token reuse detection. When a valid refresh token is presented, issue a new token pair and mark the old refresh token as revoked. If a revoked token is presented (indicating theft or replay attack), revoke the entire token chain and log a security event to the audit trail.

## Context

Refresh token rotation is a critical security measure to limit the window of vulnerability if a token is stolen. UniFlow rotates refresh tokens on every use, creating a chain of single-use tokens. If an attacker steals a token and uses it after the legitimate user has already rotated it, the reuse is detected, the entire chain is revoked, and the user must re-link the device (BR-25). This provides automatic breach detection without manual intervention.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /auth/tokens/refresh accepts JSON payload with refresh_token field
- [ ] Endpoint validates that the refresh token exists, is not expired, and is not revoked
- [ ] New access token issued with 1-hour expiry
- [ ] New refresh token issued with 90-day expiry, same chain_id, incremented sequence_num
- [ ] Old refresh token marked as revoked (revoked_at timestamp updated)
- [ ] Token rotation is atomic (transaction ensures old token revoked before new token returned)
- [ ] Reuse detection: if a revoked token is presented, entire token chain is revoked (all tokens with matching chain_id)
- [ ] Reuse detection triggers audit log entry with event_type REFRESH_REUSE_DETECTED, including source_ip
- [ ] Response returns new access_token and refresh_token on success (200)
- [ ] Response returns 401 if refresh_token is invalid or expired
- [ ] Response returns 403 if refresh_token has been revoked (reuse detected)
- [ ] Device status automatically set to OFFLINE when token chain is revoked due to reuse

## Business Rules

- BR-25: Immutable Audit Trail

## BDD Scenarios

- Refresh token reuse is logged as security event
- Expired STS token refreshed silently during resume (edge-side scenario, but relies on this endpoint)

## Technical Notes

The refresh token database record must store chain_id (UUID identifying the token family) and sequence_num (integer incremented on each rotation). Reuse detection query must find all refresh_tokens with matching chain_id and revoke them atomically. The response to a reuse attempt must be indistinguishable from an invalid token response to avoid leaking information to attackers. Edge agents must handle 403 responses by deleting local tokens and prompting the Contributor to re-link the device.
