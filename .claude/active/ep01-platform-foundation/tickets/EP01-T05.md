# EP01-T05: Agent State Reporting & Disconnect Detection

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** POST /edge/state
**Depends On:** EP01-T01, EP01-T02, EP01-T03, EP01-T04

## Summary

Implement agent state reporting endpoint where edge agents report ONLINE/OFFLINE status changes and associated metadata. Also implement a background task that checks for stale agents every 30 seconds and marks them OFFLINE if they have not polled in 110 seconds (2 missed poll cycles), enabling ungraceful disconnect detection.

## Context

Curators need timely visibility into which devices are available for upload requests. Graceful disconnects (app shutdown, user logout) send an explicit OFFLINE state change. Ungraceful disconnects (laptop crash, network loss, force-quit) must be detected server-side by checking for missed long-poll cycles (BR-11). A 2-minute detection window is acceptable for MVP — faster detection would require more aggressive polling that increases server load.

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.1

## Acceptance Criteria

- [ ] POST /edge/state accepts JSON payload with status (ONLINE or OFFLINE) and optional metadata object
- [ ] Endpoint updates device record with new status and last_seen_at timestamp
- [ ] Response returns acknowledgment on success (200)
- [ ] Background task runs every 30 seconds to check for stale agents
- [ ] Stale agent detection query finds devices with status ONLINE and last_seen_at older than 110 seconds
- [ ] Stale agents are marked OFFLINE (status updated, last_seen_at preserved for audit)
- [ ] Stale detection only affects agents not currently holding a long-poll connection
- [ ] Metadata field can include agent version, available disk space, and CPU/memory stats for admin visibility
- [ ] Endpoint validates JWT token and ensures agent_id in token matches device record
- [ ] Rate limiting does not block legitimate state changes during reconnection floods

## Business Rules

- BR-11: Ungraceful Disconnect Detection

## BDD Scenarios

- Agent crash detected within 120 seconds
- Agent network loss detected within 120 seconds
- Database writes occur only on state changes (idle agents produce zero writes)

## Technical Notes

The background task must use asyncio.create_task or a scheduled task runner (not cron). The stale detection query must exclude agents with an active long-poll connection (tracked in memory by the SignalingService, see EP01-T06). The 110-second threshold accounts for up to 55 seconds of long-poll hold plus 55 seconds of the next cycle — missing both cycles indicates a disconnect. Metadata should be stored as JSONB for flexible querying without schema changes.
