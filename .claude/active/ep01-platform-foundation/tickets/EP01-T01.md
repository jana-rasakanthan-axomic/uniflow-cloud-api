# EP01-T01: Cloud Database Schema & Migrations

**Epic:** EP-01 Platform Foundation
**Points:** 3
**Primary Endpoint:** N/A (infrastructure)
**Depends On:** None

## Summary

Establish the cloud PostgreSQL database schema with all tables required for organizations, users, devices, assets, collections, jobs, commands, and audit logging. Implement Alembic-based migrations, connection pooling configuration, and INSERT-only audit role enforcement to satisfy immutability requirements.

## Context

The cloud database is the single source of truth for all UniFlow metadata (not raw files — those go directly to S3 via BR-01). The schema must support hierarchical asset tracking (users → devices → folders → assets) with collection-level project binding (BR-24). The audit log must be immutable via database role permissions (BR-25), and the connection pool must prevent exhaustion under load (BR-06).

**Technical Design Reference:** See docs/design/platform-foundation.md Section 3.4

## Acceptance Criteria

- [ ] All 13 tables created with proper foreign key constraints (organizations, users, devices, setup_codes, folders, assets, collections, collection_assets, jobs, job_files, commands, refresh_tokens, audit_log)
- [ ] Job status CHECK constraint enforces 10 valid states (PRE_REGISTERING, WAITING_FOR_AGENT, IN_PROGRESS, PAUSED_USER, COMPLETED, PARTIALLY_FAILED, FAILED, CANCELLED, DENIED, TIMEOUT)
- [ ] File status CHECK constraint enforces 8 valid states (DISCOVERED, PRE_REGISTERED, UPLOADING, PAUSED, PAUSED_USER, SYNCED, FAILED, CANCELLED)
- [ ] Command status CHECK constraint enforces 3 valid states (PENDING, DELIVERED, EXPIRED)
- [ ] Alembic migration framework configured with version control and rollback capability
- [ ] SQLAlchemy connection pool configured with bounded max_overflow to prevent connection exhaustion
- [ ] Database role `uniflow_audit_reader` created with SELECT-only permission on audit_log (no UPDATE or DELETE grants)
- [ ] Application role has INSERT permission on audit_log but no UPDATE or DELETE permission
- [ ] RDS encryption at rest enabled (AES-256)
- [ ] Indexes created on frequently queried columns (agent_id, org_id, job status, device status, created_at timestamps)

## Business Rules

- BR-06: Persistent Database Connection Pool
- BR-24: Hierarchical Data Model with Collection-Level Project Binding
- BR-25: Immutable Audit Trail

## BDD Scenarios

- Cloud database contains only metadata and thumbnail references
- Database writes occur only on state changes
- Cloud broker maintains bounded database connections
- Assets from one folder assigned to different projects
- Collection exists without project binding during curation
- Device linking is logged in audit trail
- Failed link attempts are logged
- Refresh token reuse is logged as security event

## Technical Notes

The schema must enforce hierarchical relationships: organizations own users, users own devices, devices own folders, folders own assets. Collections are a cross-cutting concern that reference assets across multiple folders. Use BIGSERIAL for audit_log.id to support high-volume logging. Enable PostgreSQL CHECK constraints for state machines as defense-in-depth alongside application-level validation. Use connection pool settings that accommodate thousands of held long-poll connections without proportional database connection consumption.
